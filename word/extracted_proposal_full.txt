☑教学实践（含实验） □其它一、研究的背景和意义1.1 研究背景随着互联网技术的快速发展和数字经济的不断深化，电子商务已成为现代商业体系中的重要组成部分。在激烈的市场竞争环境下，企业逐渐由“经验驱动”向“数据驱动”转型，精细化运营成为电商平台提升用户转化率与商业收益的核心手段。其中，广告投放作为电商平台获取流量和促进交易的重要方式，其投放效果评估与策略优化高度依赖对海量数据的高效处理与分析能力。在电商广告业务中，系统每天会产生大量结构化与半结构化数据，包括用户行为日志（曝光、点击、加购、下单等）、广告投放日志（广告 ID、出价、创意信息、人群定向条件等）以及业务结果数据。这些数据具有数据规模大、写入频繁、更新比例高、时效性要求强等特点，对底层数据架构和计算框架提出了更高要求。传统的数据处理方式通常采用将原始日志存储在 HDFS 或对象存储中，再使用 MapReduce、Hive 等工具进行离线批处理分析。然而，这种架构在实际应用中逐渐暴露出诸多问题：（1）数据更新与删除成本高：传统列式存储格式（如 Parquet、ORC）对数据更新和删除支持较弱，当需要修正历史数据时，往往需要重写整个分区甚至全表，效率低下；（2）缺乏高效的增量处理机制：下游分析任务难以只消费新增或变更数据，通常需要进行全量扫描，造成资源浪费和任务延迟；（3）事务与一致性保障不足：在多任务并发写入场景下，难以有效保证数据的原子性和一致性；（4）数据治理与管理复杂：元数据分散、数据血缘关系不清晰，数据质量与可维护性难以保障。与此同时，业务对实时数据分析能力的需求日益增强，例如实时监控广告曝光量、点击率和转化效果。为同时满足离线分析与实时计算需求，业界广泛采用 Lambda 架构，即并行构建批处理链路与流处理链路。尽管该架构能够在一定程度上解决时效性问题，但也带来了技术栈割裂、开发逻辑重复、数据口径不一致、系统维护成本高等新的挑战。在此背景下，数据湖（Data Lake）与湖仓一体（Lake house）架构逐渐兴起。Apache Hudi（Hadoop Upserts Deletes and Incrementals）作为数据湖领域的重要开源组件，通过引入 ACID 事务机制、时间轴管理、增量数据消费以及多种表类型（Copy on Write、Merge on Read），有效弥补了传统数据湖在数据更新、增量处理和一致性方面的不足，为构建统一的离线与实时分析平台提供了基础。基于 Apache Hudi 的统一存储层，可以结合 Apache Flink 实现低延迟的实时流处理，结合 Apache Spark 实现高吞吐的离线批处理与多维分析。通过共享同一份底层数据，实时与离线任务在存储层实现统一，有效降低系统复杂性，提升数据一致性和整体处理效率。因此，围绕 Apache Hudi 构建一个面向电商广告场景的实时与离线一体化分析系统，具有较高的研究价值和实践意义。1.2 研究意义本研究以电商广告业务为应用场景，设计并实现一个基于 Apache Hudi 的实时与离线融合分析系统，其研究意义主要体现在以下几个方面：（一）理论意义（1）湖仓一体架构的实践探索：通过在实际业务场景中引入 Apache Hudi，探索湖仓一体架构在电商广告数据处理中的应用模式与实现路径，总结其在数据更新、增量处理和一致性保障方面的优势与不足，为相关理论研究和工程实践提供参考案例。（2）批流融合处理模式的研究：依托 Hudi 的统一存储与增量消费能力，研究如何将 Apache Flink 的实时流处理与 Apache Spark 的离线批处理进行工程化融合，形成“存储统一、计算分层”的数据处理范式，为批流融合技术提供可落地的实现方案。（3）新型大数据技术协同价值验证：通过系统设计与实现，验证 Apache Hudi、Apache Spark、Apache Flink 等新兴大数据技术在解决传统数据架构痛点方面的实际效果，分析其协同工作的可行性与性能表现。（二）实践意义（1）提升数据处理效率与一致性通过构建基于 Hudi 的统一湖仓存储层，实现实时与离线数据的集中管理，解决传统文件存储方式下数据更新困难、增量处理效率低和一致性保障不足等问题，提升整体数据处理效率。（2）降低系统复杂度与运维成本统一的存储层减少了数据冗余和多链路同步成本，简化了数据架构设计。实时与离线计算共享同一数据源，有助于降低系统维护难度和开发成本。（3）支撑电商广告精细化运营系统既能够提供低延迟的实时监控指标（如实时曝光量、点击量、转化率），又能够支持高准确度的离线分析（如广告效果评估、用户行为分析和长期趋势分析），为广告投放优化和运营决策提供全面的数据支持。（4）为相关行业提供参考价值本研究形成的系统架构设计与实现经验，可为其他具有类似数据处理需求的互联网业务场景（如内容推荐、在线教育、游戏运营等）提供借鉴。二、文献综述围绕电商广告数据分析系统的研究，学术界与工业界的相关工作主要集中在广告日志数据处理架构、实时与离线计算体系、数据仓库建模方法以及湖仓一体技术等方面。随着电商业务规模和数据体量的不断增长，传统以离线批处理为主的数据处理模式逐渐难以满足广告分析在时效性、一致性和可维护性方面的要求。2.1 电商广告数据分析系统相关研究电商广告系统在运行过程中会持续产生大量用户行为数据和广告投放数据，如广告曝光、点击等事件日志。这类数据具有写入频繁、规模大、时间序列特征明显等特点，是广告效果分析和运营决策的重要数据基础。早期广告数据分析系统多采用基于日志采集与离线统计的方式，通过周期性汇总曝光量、点击量等指标为业务提供支持。随着数据规模增长，基于分布式存储和计算框架的数据分析方案逐渐成为主流，HDFS、Hive 等技术被广泛应用于广告日志的离线分析场景中。2.2 实时与离线数据处理架构研究为提升数据分析的时效性，实时计算框架逐渐被引入广告数据处理系统。Apache Kafka 作为高吞吐、低延迟的消息队列系统，被广泛用于日志数据的实时传输；Apache Flume 则常用于从业务系统中采集日志数据并写入分布式存储或消息系统 。在实际工程中，广告数据分析系统通常同时包含离线批处理与实时流处理两类任务。Apache Spark 以其成熟的批处理和 SQL 分析能力，被广泛应用于离线广告数据统计与数据仓库构建；Apache Flink 则凭借低延迟和状态一致性优势，被用于实时指标计算和监控分析。然而，传统架构下实时与离线链路往往相互独立，系统复杂度较高。2.3 数据湖与湖仓一体技术研究数据湖作为一种灵活的数据存储模式，能够支持多种数据类型和计算方式，但传统数据湖在数据更新和一致性管理方面存在不足。为此，湖仓一体（Lake house）架构应运而生，通过融合数据湖的灵活性和数据仓库的管理能力，为分析型应用提供统一的数据基础。Apache Hudi 是湖仓一体架构的重要实现之一。Hudi 通过引入时间轴管理和事务机制，支持数据的增量写入、更新和删除，并提供增量查询能力，能够有效支撑日志型和事件型数据的分析需求。2.4 基于统一湖仓的广告数据分析研究随着 Apache Hudi 对 Apache Spark 和 Apache Flink 的原生支持不断增强，基于统一湖仓存储的实时与离线融合分析架构逐渐成熟。在该架构下，实时处理任务将数据写入 Hudi，离线分析任务基于同一数据源进行统计分析，从而降低数据冗余并提升数据一致性。在广告数据分析场景中，该模式能够同时满足实时监控和离线分析的需求，具有良好的工程实践价值。三、研究的主要内容和方法3.1 研究的主要内容本项目以电商广告业务为应用背景，围绕广告日志数据的采集、存储、处理与分析，设计并实现一个基于 Apache Hudi 的实时与离线一体化分析系统。研究内容主要包括以下几个方面：（1）电商广告数据采集与数据源构建以阿里云天池公开数据集 Ali_Display_Ad_Click 为基础，结合电商广告业务数据特点，对广告曝光、点击等行为日志进行整理和建模。研究广告数据在真实业务中的产生方式和数据结构，模拟日志采集流程，为后续实时与离线分析提供统一的数据输入。在系统设计中，采用消息队列实现数据的缓冲与解耦，为实时处理和离线处理提供稳定的数据来源。（2）基于 Apache Hudi 的统一湖仓存储层设计围绕湖仓一体架构思想，研究 Apache Hudi 在广告日志数据场景下的应用方式，设计统一的数据存储层。通过合理选择 Hudi 表类型和分区策略，实现广告数据的高效写入、更新和历史数据管理。在该统一存储层中，实时数据和离线数据共享同一份底层数据，实现数据的一致存储和统一管理，为后续多种计算任务提供可靠的数据基础。（3）实时广告数据处理与分析模块设计针对广告业务对实时指标的需求，设计实时数据处理流程。通过流式计算框架对实时广告日志进行处理，完成曝光量、点击量等核心指标的实时统计，并将计算结果持续写入 Hudi 存储层，实现实时数据与历史数据的无缝衔接。该模块重点研究实时数据处理流程的稳定性和一致性，保证实时统计结果能够与离线分析口径保持一致。（4）离线数据仓库构建与多维分析基于统一湖仓存储层，采用离线计算引擎对广告数据进行分析，构建分层数据仓库体系。通过对原始广告日志数据进行清洗、关联和聚合，形成适用于广告分析的明细层、汇总层和应用层数据。在此基础上，实现按广告、用户、资源位、时间等多维度的离线统计分析，为广告效果评估和业务决策提供数据支持。（5）分析结果服务化与可视化展示将离线分析和实时统计的结果数据同步至关系型数据库，对外提供统一的数据访问接口。通过可视化工具和 Web 系统展示广告分析结果，实现用户登录、指标查询和报表展示等功能，完成从数据采集到分析结果展示的完整闭环。3.2 研究方法为完成上述研究内容，本项目主要采用以下研究方法：（1）系统架构设计方法在充分分析电商广告数据特点和业务需求的基础上，采用分层设计思想，对系统整体架构进行规划。通过将数据采集层、存储层、计算层和应用层进行解耦设计，提高系统的可扩展性和可维护性。（2）工程实现与原型验证方法基于成熟的开源大数据技术框架，对系统关键模块进行工程实现。通过搭建实验环境，对数据采集、实时处理、离线分析和结果展示等功能进行实现和验证，确保系统方案具有可行性和稳定性。（3）批流融合处理方法依托 Apache Hudi 的统一存储能力，研究实时流处理与离线批处理在同一数据源上的协同方式。通过实时写入与离线分析相结合的方式，实现广告数据的实时统计与历史分析融合处理，避免传统架构中多链路带来的数据不一致问题。（4）对比分析与验证方法通过对系统运行结果进行分析，验证所设计架构在数据处理效率、数据一致性和系统复杂度等方面的表现。对比传统离线分析方式与统一湖仓架构下的处理流程，评估本项目方案在实际广告分析场景中的应用价值。四、主要参考文献阿里云天池.
